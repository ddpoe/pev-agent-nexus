# Test Annotation Policy

## Decision Rule

**"Would a non-developer stakeholder recognize this test as a product story?"**

- Yes → Tier 3 (full annotation)
- No, but it tests a meaningful subsystem → Tier 2 (purpose only)
- No, it's exercising internal logic → Tier 1 (plain pytest)

## Tier 1 — Plain pytest

No `@workflow`, no `Step()`. Good module and function docstrings are sufficient.

These tests change frequently as internals evolve. Decorating them creates maintenance friction without proportional documentation value. Unit tests for individual functions, edge cases, and internal helpers live here.

## Tier 2 — @workflow(purpose=...)

Purpose string only — no Steps. The purpose gives cortex something to index and display in the viz, and signals intent to a reader without requiring a full narrative.

Use this for tests that exercise a meaningful subsystem (e.g. the full scanner pipeline, the builder edge-filtering logic) but aren't stakeholder-readable stories.

```python
@workflow(purpose="Verify that scan_module emits validates edges for all direct calls in a test function")
def test_validates_edges_from_direct_calls(tmp_path):
    ...
```

## Tier 3 — @workflow(purpose=...) + Step()/AutoStep()

Full `Step()` markers narrate the test so a reader never needs to open the source. Use this for E2E tests and user-story-level scenarios.

`Step()` markers are the specification. Use `critical="NOT IMPLEMENTED"` on skeleton steps to document the roadmap without faking passing tests.

```python
@workflow(
    purpose="Verify that cortex build auto-generates validates edges from test call graph with no manual annotation",
)
def test_ast_validates_edges_end_to_end(tmp_path):
    口 = Step(step_num=1, name="Write production module",
              purpose="Create a .py file with an indexed function")
    ...

    口 = Step(step_num=2, name="Write test file",
              purpose="Create a test_*.py that imports and calls the production function")
    ...

    口 = Step(step_num=3, name="Run cortex build",
              purpose="Scan both files and upsert nodes and edges")
    ...

    口 = Step(step_num=4, name="Assert validates edge exists",
              purpose="Confirm the test function has a validates edge to the production function")
    ...
```

## Coverage and the AST Call Graph

**Do not use `covers=[]`.** Coverage links are auto-generated by cortex's AST call graph scanner at build time. Any function call from a test function to an indexed production function automatically produces a `validates` edge — no annotation required.

The `@workflow` decorator is used for its `purpose` and `Step()` narrative only. It has no role in coverage mapping.

## dFlow Note

`@workflow` on a test function is detected by dFlow as a workflow entry point (not distinguished from production workflows in `dflow list`). This is an accepted tradeoff — cortex tags test functions by file convention (`test_*.py`) regardless of decorator, so the semantic distinction is preserved in the cortex index.

## Test Budget

**Guideline: 5-10 focused tests per subsystem change.** Past 15, you are likely testing implementation details rather than behavior — consolidate.

The goal is tests that each cover a meaningful scenario. Three similar assertions about the same behavior belong in one test, not three. A test that exercises a complete flow (write data → read data → verify) is worth more than five tests that each check one field.

**Signs of over-testing:**
- Multiple tests that differ only in the input value (parameterize instead)
- Separate tests for setup, action, and assertion that could be one test
- Tests that duplicate assertions already covered by another test at a higher level
- Tests for internal helpers that are already exercised through the public API test

**Signs of under-testing:**
- Changed code nodes with no `validates` edge (no test covers them)
- Edge cases mentioned in the design spec but not exercised
- Error paths with no negative test

**Auditor enforcement:** The Auditor references this section during the test audit. Excessive test counts go in the Impact Report under `needs_fix` with category `test_budget`, along with specific consolidation recommendations. The Builder is expected to generate tests within budget; the Auditor catches overruns.
